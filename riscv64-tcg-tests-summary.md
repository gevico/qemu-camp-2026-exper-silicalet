# RISC-V64 TCG Tests Summary

本文档基于 `tests/gevico/tcg/riscv64/test-insn-*.c` 源码逐一整理，目标不是简单复述文件名，而是说明：

- 每道题实际在测哪条自定义指令
- 指令在测试里被当作什么语义来验证
- 测试函数分别覆盖了哪些输入和边界
- 这套用例已经覆盖了什么、还没有覆盖什么

## 1. 测试框架的共同结构

所有题都建立在同一套最小运行时上：

- 头文件：`tests/gevico/tcg/riscv64/crt/crt.h`
- 核心辅助：
  - `printf(...)`：打印失败信息
  - `memset(...)` / `memcpy(...)`：准备测试数据
  - `crt_assert(condition)`：条件失败就打印文件和行号并 `crt_abort()`

所有测试文件都遵循同一个模式：

1. 用内联汇编 `.insn r 0x7b, 6, <funct7>, ...` 封装一条自定义指令。
2. 用 C 写出“软件参考实现”。
3. 运行硬件/TCG 结果和软件结果。
4. 逐项比较；一旦不一致立即失败。
5. 部分测试还会加一个额外 `crt_assert(...)` 做 spot-check。

因此，这些题本质上测的是：

> QEMU/TCG 对 opcode `0x7b`、`funct3 = 6` 这一组自定义指令的行为实现，是否与测试作者预期的软件语义一致。

## 3. 逐题说明

---

### 3.1 `test-insn-dma.c`

#### 源码中的指令接口

- 封装函数：`custom_dma(float *dst, const float *src, long grain)`
- 指令编码：`.insn r 0x7b, 6, 6`

#### 这题把指令当成什么来测

这题把 `dma` 指令视为“`float32` 方阵转置”。

软件参考实现是：

```c
dst[j * n + i] = src[i * n + j];
```

也就是把一个按行主序存储的 `n x n` 矩阵转置后写到 `dst`。

#### 具体测了哪些 case

1. `test_dma_grain_8x8()`
   - `n = 8`
   - 输入矩阵为 `0.0f, 1.0f, 2.0f, ...`
   - 调用：`custom_dma(dst_hw, src, 0)`
   - 预期：`grain = 0` 对应 `8x8` 转置

2. `test_dma_grain_16x16()`
   - `n = 16`
   - 调用：`custom_dma(dst_hw, src, 1)`
   - 预期：`grain = 1` 对应 `16x16` 转置

3. `test_dma_grain_32x32()`
   - `n = 32`
   - 调用：`custom_dma(dst_hw, src, 2)`
   - 预期：`grain = 2` 对应 `32x32` 转置

#### 关键验证点

- `grain` 不是任意值，而是在测试中被解释为矩阵规模选择器：
  - `0 -> 8x8`
  - `1 -> 16x16`
  - `2 -> 32x32`
- 比较函数不是直接做浮点比较，而是把 `float *` 强转成 `uint32_t *` 后逐项比较位模式。
- 因此这里测的是“转置后的每个 `float32` bit pattern 完全一致”，不是近似数值相等。

#### 实际覆盖到的行为

- 行列下标交换是否正确
- 不同规模矩阵下寻址是否正确
- 输出缓冲区写入位置是否正确

#### 没有覆盖到的点

- 非方阵
- `grain` 取非法值
- 原地转置 `dst == src`
- 特殊浮点值，如 `NaN`、`Inf`、`-0.0`

### 3.2 `test-insn-gemm.c`

#### 源码中的指令接口

- 封装函数：`custom_gemm(int *c, const int *a, const int *b)`
- 指令编码：`.insn r 0x7b, 6, 14`

#### 这题把指令当成什么来测

这题把 `gemm` 指令视为“`4x4 int32` 矩阵乘法”。

软件参考实现是标准三重循环：

```c
for (i)
  for (j) {
    long acc = 0;
    for (k)
      acc += (long)a[i * 4 + k] * (long)b[k * 4 + j];
    c[i * 4 + j] = (int)acc;
  }
```

也就是说：

- 输入矩阵大小固定为 `4x4`
- 元素类型是 `int32`
- 中间累加使用 `long`
- 最终再截断回 `int`

#### 具体测了哪些 case

1. `test_gemm_basic()`
   - `A = {1..16}`
   - `B = {17..32}`
   - 比较整个 16 元素结果矩阵
   - 额外断言 `C[0][0] == 250`

2. `test_gemm_identity()`
   - `A = {1..16}`
   - `B = I`，即 `4x4` 单位矩阵
   - 验证 `A * I == A`

#### 关键验证点

- 基础矩阵乘法是否正确
- `k` 维累加是否正确
- 输出矩阵元素落点是否正确
- 至少覆盖了一个代数恒等性质：右乘单位矩阵不变

#### 没有覆盖到的点

- 左乘单位矩阵 `I * A`
- 全零矩阵、负数矩阵、随机矩阵
- 溢出或大数累加行为
- `c` 与 `a` / `b` 重叠时的行为

### 3.3 `test-insn-sort.c`

#### 源码中的指令接口

- 封装函数：`custom_sort(long k, const int *arr, long n)`
- 指令编码：`.insn r 0x7b, 6, 22`

注意：函数声明里 `arr` 是 `const int *`，但这条指令显然会修改该数组内容，所以这里的 `const` 只是 C 接口写法上的不严谨，不能按“不写内存”理解。

#### 这题把指令当成什么来测

这题把 `sort` 指令视为：

> 对数组前 `k` 个元素进行升序排序，其余元素保持原样。

软件参考实现是：

```c
bubble_sort(arr, k);
```

其中 `bubble_sort` 只遍历前 `k` 个元素，后半段不参与排序。

#### 具体测了哪些 case

- 数组总长度固定 `ARRAY_SIZE = 32`
- 排序长度固定 `SORT_LEN = 16`
- 初始数组是固定的 32 个整数
- 调用：
  - `custom_sort(16, hw_arr, 32)`
  - `bubble_sort(sw_arr, 16)`
- 最终比较整个 32 元素数组

#### 关键验证点

这题实际上同时验证了两件事：

1. 前 16 个元素是否被正确排成升序
2. 后 16 个元素是否保持原值，没有被错误写坏

因为参考实现只排序前缀，而最终 `compare()` 会比较整段 32 个元素。

#### 没有覆盖到的点

- `k = 0`、`k = 1`、`k = n` 等边界
- 重复元素
- 已有序、逆序、全相同
- 负数输入
- 稳定性

### 3.4 `test-insn-vadd.c`

#### 源码中的指令接口

- 封装函数：`custom_vadd(int *c, const int *a, const int *b)`
- 指令编码：`.insn r 0x7b, 6, 30`
- 向量长度固定：`VEC_LEN = 16`

#### 这题把指令当成什么来测

这题把 `vadd` 指令视为：

> 两个长度固定为 16 的 `int32` 向量做逐元素加法。

软件参考实现是：

```c
c[i] = a[i] + b[i];
```

#### 具体测了哪些 case

1. `test_vadd_basic()`
   - `a = {1,2,...,16}`
   - `b = {100,200,...,1600}`
   - 逐项比较硬件/软件结果
   - 额外断言：
     - `dst_sw[0] == 101`
     - `dst_sw[1] == 202`

2. `test_vadd_overflow()`
   - `a[i] = INT32_MAX - i`
   - `b[i] = i + 1`
   - 用于覆盖接近 `int32` 上界的输入

3. `test_vadd_inplace()`
   - 先用软件实现算出参考结果
   - 再调用 `custom_vadd(a, a, b)`
   - 也就是目标数组和第一个源数组重合

#### 关键验证点

- 正常逐元素相加
- 输出数组每个元素位置是否正确
- 原地写回 `dst == src1` 是否正确
- 接近 `INT32_MAX` 的数据下是否仍与参考实现一致

#### 关于“overflow”这组数据的说明

这组数据中：

- `a[0] = INT32_MAX`
- `b[0] = 1`

从纯 ISO C 语义讲，`signed int` 溢出是未定义行为；但测试作者显然是希望它表达“按目标实现的 `int32` 加法结果去比对”。换句话说，这个 case 反映的是测试作者对目标机器算术行为的预期，而不是严格的可移植 C 规范。

#### 没有覆盖到的点

- `dst == b` 的原地写回
- 长度不是 16 的情况
- 全负数、全零、随机输入

### 3.5 `test-insn-crush.c`

#### 源码中的指令接口

- 封装函数：`custom_crush(uint8_t *dst, const uint8_t *src, long n)`
- 指令编码：`.insn r 0x7b, 6, 38`

#### 这题把指令当成什么来测

这题把 `crush` 指令视为：

> 把输入字节流中每两个元素的低 4 bit 打包成一个输出字节。

软件参考实现：

```c
dst[i]  = src[2*i] & 0x0F;
dst[i] |= (src[2*i+1] & 0x0F) << 4;
```

也就是：

- 偶数位输入的低 4 bit -> 输出低 nibble
- 奇数位输入的低 4 bit -> 输出高 nibble

#### 具体测了哪些 case

- 输入长度：`SRC_LEN = 10`
- 输出长度：`DST_LEN = 5`
- 固定输入：
  - `0x0A, 0x0B, 0x0C, 0x0D, 0x0E, 0x0F, 0x01, 0x02, 0x03, 0x04`
- 调用：`custom_crush(dst_hw, src_data, 10)`

按照软件模型，预期输出为：

- `dst[0] = 0xBA`
- `dst[1] = 0xDC`
- `dst[2] = 0xFE`
- `dst[3] = 0x21`
- `dst[4] = 0x43`

#### 关键验证点

- 每一对输入字节是否正确组合成一个输出字节
- 高/低 nibble 的排列顺序是否正确
- 是否只取低 4 bit

#### 代码里顺带暴露的一点

`pack_low4bits()` 其实还写了“奇数长度时最后一个元素单独保留低 4 bit”的逻辑：

```c
if (n & 1)
    dst[out_len - 1] = src[n - 1] & 0x0F;
```

但当前测试传入的是 `n = 10`，是偶数，所以这个分支根本没有被覆盖到。

#### 没有覆盖到的点

- `n` 为奇数
- 输入字节高 4 bit 非零时是否被完全忽略
- `dst == src` 重叠内存

### 3.6 `test-insn-expand.c`

#### 源码中的指令接口

- 封装函数：`custom_expand(uint8_t *dst, const uint8_t *src, long n)`
- 指令编码：`.insn r 0x7b, 6, 54`

#### 这题把指令当成什么来测

这题把 `expand` 指令视为：

> 把每个输入字节拆成两个 4 bit 元素，输出长度翻倍。

软件参考实现：

```c
dst[2*i]     = src[i] & 0x0F;
dst[2*i + 1] = (src[i] >> 4) & 0x0F;
```

注意顺序是：

- 先放低 4 bit
- 再放高 4 bit

#### 具体测了哪些 case

- 输入长度：`SRC_LEN = 5`
- 输出长度：`DST_LEN = 10`
- 固定输入：`0xAB, 0xCD, 0xEF, 0x12, 0x34`

按参考实现，预期输出应为：

- `0xAB -> 0x0B, 0x0A`
- `0xCD -> 0x0D, 0x0C`
- `0xEF -> 0x0F, 0x0E`
- `0x12 -> 0x02, 0x01`
- `0x34 -> 0x04, 0x03`

合起来就是：

`{0x0B, 0x0A, 0x0D, 0x0C, 0x0F, 0x0E, 0x02, 0x01, 0x04, 0x03}`

#### 关键验证点

- 每个输入字节是否被拆成两个输出元素
- 低位/高位顺序是否符合测试定义
- 输出地址步进是否正确

#### 与 `crush` 的关系

从测试作者给的软件模型看：

- `expand` 是“1 字节拆成两个 nibble”
- `crush` 是“两个 nibble 再压回 1 字节”

二者在测试意图上是一对互逆方向的变换。

#### 没有覆盖到的点

- 空输入
- 输出缓冲区与输入缓冲区重叠
- 任意长度随机输入

### 3.7 `test-insn-vdot.c`

#### 源码中的指令接口

- 封装函数：`custom_vdot(const int *a, const int *b) -> long`
- 指令编码：`.insn r 0x7b, 6, 70`
- 向量长度固定：`VEC_LEN = 16`

#### 这题把指令当成什么来测

这题把 `vdot` 指令视为：

> 两个 16 元素 `int32` 向量做点积，并返回 64 位结果。

软件参考实现：

```c
long acc = 0;
for (i = 0; i < 16; i++)
    acc += (long)a[i] * (long)b[i];
```

#### 具体测了哪些 case

只有一个测试函数：`test_vdot_basic()`

- `A = {1,2,3,...,16}`
- `B = {16,15,14,...,1}`
- 计算并比较硬件结果与软件结果
- 再做一个常量断言：`sw == 816`

#### 为什么结果是 816

因为：

```text
1*16 + 2*15 + ... + 16*1 = 816
```

这个 spot-check 可以防止“软件参考实现和硬件实现同时写错但恰好一致”的低级失误。

#### 关键验证点

- 逐元素乘法是否正确
- 累加顺序对应的结果是否正确
- 返回寄存器是否能承载 `long` 结果

#### 没有覆盖到的点

- 负数输入
- 大数乘加导致的边界
- 全零向量
- 不同长度

---

### 3.8 `test-insn-vrelu.c`

#### 源码中的指令接口

- 封装函数：`custom_vrelu(int *dst, const int *src, long n)`
- 指令编码：`.insn r 0x7b, 6, 86`
- 向量长度：`VEC_LEN = 16`

#### 这题把指令当成什么来测

这题把 `vrelu` 指令视为标准 ReLU：

```c
dst[i] = (src[i] > 0) ? src[i] : 0;
```

即：

- 正数保留
- 负数清零
- 0 保持 0

#### 具体测了哪些 case

1. `test_vrelu_mixed()`
   - 固定输入：
     `{-5, 3, -1, 0, 7, -100, 42, -3, 8, -9, 15, -20, 1, 0, -7, 99}`
   - 覆盖三类值：负数、正数、0
   - 先把 `dst_hw` 和 `dst_sw` 清零，再分别运行

2. `test_vrelu_inplace()`
   - 先把 `src_data` 拷贝到本地数组 `inplace`
   - 调用：`custom_vrelu(inplace, inplace, 16)`
   - 即 `dst == src`

#### 关键验证点

- ReLU 三种分支是否都正确
- 目标数组和源数组重合时，是否还能得到正确结果

#### 这个测试里有一个隐含前提

`test_vrelu_inplace()` 直接拿 `dst_sw` 做参考结果比较，但它自己没有重新计算软件参考，而是依赖前一个 `test_vrelu_mixed()` 已经把 `dst_sw` 写成正确答案。

也就是说，这个文件还隐含测试了：

- `main()` 必须先跑 `test_vrelu_mixed()`
- 再跑 `test_vrelu_inplace()`

如果单独抽出第二个函数执行，它并不是自洽的。

#### 没有覆盖到的点

- 全正 / 全负 / 全零输入
- `n < 16`
- 很大正数或很小负数

---

### 3.9 `test-insn-vscale.c`

#### 源码中的指令接口

- 封装函数：`custom_vscale(int *dst, const int *src, long scale)`
- 指令编码：`.insn r 0x7b, 6, 102`
- 向量长度固定：`VEC_LEN = 16`

#### 这题把指令当成什么来测

这题把 `vscale` 指令视为：

> 长度为 16 的 `int32` 向量，每个元素乘同一个标量 `scale`。

软件参考实现：

```c
dst[i] = (int)((long)src[i] * scale);
```

即中间先扩到 `long` 再回写为 `int`。

#### 具体测了哪些 case

1. `test_vscale_basic()`
   - 输入：`src = {1..16}`
   - 标量：`3`
   - 额外断言：
     - `dst_sw[0] == 3`
     - `dst_sw[15] == 48`

2. `test_vscale_negative()`
   - 输入含正负交替值：
     `{10, -20, 30, -40, ... , 150, -160}`
   - 标量：`-2`

#### 关键验证点

- 正标量缩放
- 负标量缩放
- 正负号传播是否正确
- 16 个元素是否全部按同一标量处理

#### 没有覆盖到的点

- `scale = 0`
- `scale = 1` / `-1`
- 大数乘法溢出
- 原地写回 `dst == src`

---

### 3.10 `test-insn-vmax.c`

#### 源码中的指令接口

- 封装函数：`custom_vmax(const int *src, long n) -> long`
- 指令编码：`.insn r 0x7b, 6, 118`

#### 这题把指令当成什么来测

这题把 `vmax` 指令视为：

> 对一个 `int32` 数组做最大值归约，并把结果按有符号 `int32` 符号扩展后返回到 `long`。

软件参考实现：

```c
int max = src[0];
for (i = 1; i < n; i++)
    if (src[i] > max)
        max = src[i];
return (long)(int)max;
```

测试里还专门写了注释：

> sign-extend INT32 to XLEN(64)

这说明作者明确要求返回值在 `riscv64` 上应表现为“有符号 32 位数扩展到 64 位”。

#### 具体测了哪些 case

1. `test_vmax_positive()`
   - 输入：`{5, 23, 1, 99, 42, 7, 88, 3}`
   - 预期最大值：`99`

2. `test_vmax_negative()`
   - 输入：`{-5, -23, -1, -99, -42, -7, -88, -3}`
   - 预期最大值：`-1`

#### 关键验证点

- 正数集合中的最大值选择
- 全负数集合中的最大值选择
- 返回值是否进行了正确的符号扩展

#### 没有覆盖到的点

- 重复最大值
- `n = 1`
- `INT32_MIN` / `INT32_MAX`
- 长度更大或更小的数组

## 4. 这批题共同测到的能力

如果把这些文件整体看，这套用例大致在验证以下几类自定义指令：

1. 向量算术
   - `vadd`
   - `vscale`
   - `vdot`
   - `vmax`
   - `vrelu`

2. 小型矩阵/块运算
   - `gemm`
   - `dma`（这里被定义成转置）

3. 数据重排/压缩
   - `sort`
   - `expand`
   - `crush`

## 5. 这批题的整体特点

### 5.1 优点

- 每题都有明确的软件黄金模型
- 输入数据固定，失败时容易复现
- 部分题覆盖了边界或特殊路径：
  - `vadd`：原地写回、接近上界
  - `vrelu`：原地写回
  - `vscale`：负标量
  - `vmax`：全负数
  - `sort`：只排序前缀
  - `dma`：三种规模

### 5.2 明显不足

整体上这套题是“功能冒烟测试”，不是“强覆盖测试”。共有几个共性缺口：

- 长度参数大多只测了一个固定值
- 大多数输入都是手工构造的小样本，随机性不够
- 很多题没有覆盖 `0`、空输入、最小长度、最大长度、非法参数
- 很多题没测源/目的缓冲区重叠
- 有些题没测负数或极值
- `vadd` 的 overflow case 依赖实现定义/目标机语义，不是严格可移植的 C 参考

## 6. 一句话概括每题

- `dma`：测 `float32` 方阵转置，按 `grain` 区分 `8/16/32` 阶。
- `gemm`：测 `4x4 int32` 矩阵乘法是否正确。
- `sort`：测“只对前 16 个元素排序，不动后 16 个元素”。
- `vadd`：测 16 路 `int32` 向量加法，带原地写回和接近上界输入。
- `crush`：测两个低 4 bit 是否能正确打包回一个字节。
- `expand`：测一个字节是否能按“低 nibble 在前，高 nibble 在后”拆开。
- `vdot`：测 16 路 `int32` 点积及 64 位返回值。
- `vrelu`：测 ReLU 的三路分支和原地执行。
- `vscale`：测向量按标量缩放，尤其是负标量。
- `vmax`：测最大值归约，尤其是全负数场景下的比较和符号扩展。
